# 行情数据源说明

## 目的与范围

本说明覆盖应用内的实时行情、历史日线与缺失历史同步。行情数据只保存在设备本地 IndexedDB；外部接口仅在用户触发刷新或同步时访问。

## 数据源

| 数据源 | 运行环境 | 覆盖范围 | 鉴权 | 用途 |
| --- | --- | --- | --- | --- |
| `android-default` | Android 原生应用 | A 股、港股、美股股票；美股期权 | 无 | 安卓默认的实时与历史补齐来源 |
| Tencent | `android-default` 的股票实现 | A 股、港股、美股股票 | 无 | 实时行情与复权日线 |
| Sina | `android-default` 的回退实现 | A 股、港股、美股股票 | 无 | 腾讯实时行情缺失时回退 |
| Yahoo Finance | `android-default` 的期权实现 | 美股期权 | 无 | 期权实时与历史日线 |
| Twelve Data | Web / Android 可选 | 美股、港股、A 股等 | API Key | 用户配置的云端数据源 |
| iTick | Web / Android 可选 | 美股、港股、A 股等 | API Key | 用户配置的云端数据源 |
| MarketData.app | Web / Android 可选 | 美股股票和期权 | API Key | 用户配置的云端数据源 |

`android-default` 不是独立的网络服务名称，而是安卓端对 Tencent → Sina（股票）及 Yahoo（期权）的统一适配层。它经由原生 `NativeMarket` 网络插件发出请求，因此不受 WebView CORS 限制。

腾讯历史日线单次请求最多返回 800 条；超过该限制会返回 `bad params`，并不能视为标的没有数据。

## 缺失历史同步流程

1. “检测缺失区间”按标的聚合交易记录：起点为该标的最早的可报价交易日。应用使用与持仓页一致的数量计算规则判断净仓位：仍有净持仓（多头或空头）的标的，终点为该市场本地的今天（港股为 `Asia/Hong_Kong`、美股为 `America/New_York`、A 股为 `Asia/Shanghai`）；已清仓标的，终点为最后一笔使仓位归零的可报价交易日。周末和休市日不会凭空生成 K 线，数据源会返回区间内最后一个交易日。
2. “开始同步”唤醒 `MarketTaskExecutor`。
3. 执行器按启用状态、优先级、市场和资产类型，从能力清单中选取可执行数据源。
4. 数据源返回日线后，执行器写入 `historicalBars`，更新 `historicalCoverage`，再把工作项标记为成功、部分成功或重试。

安卓首次建库会启用 `android-default`（优先级 0）；已有安卓安装在数据库升级到版本 8 时会补充此配置。该来源不需要 API Key，执行器会将它视为可用来源。Web 端保留为禁用状态，避免绕过浏览器的跨域策略。

### 历史日线的请求粒度

不是“每个交易日发一个请求”。应用先为每个标的生成一个日期区间任务；一次历史请求会要求数据源返回该标的在整个区间内的全部日 K 线，再逐条写入本地缓存。规划器可以把日期范围相同的标的放进一个计划（最多 8 个），但当前执行器仍会顺序调用各标的一次历史接口。因此当前安卓默认来源的实际网络粒度是“每个标的、每个区间一次请求”，而不是“每根日 K 一次请求”。腾讯单个请求最多 800 条日线。

## 排查“点击开始同步后没有数据”

优先在“行情缓存管理”页确认：

- 缺失检测后是否生成了待同步任务；
- 任务是否进入 `running`、`retry_scheduled`、`paused_quota` 或 `failed_permanent` 状态；
- 对于安卓端，`android-default` 是否已随本版本迁移写入配置；
- 标的市场与资产类型是否受所选数据源支持。

本次修复前，安卓默认来源虽已实现网络抓取，但未注册到执行器、能力清单和数据库配置中；同时执行器要求数据源必须有 API Key。结果是历史补齐无法生成安卓可执行计划。现已将免密来源接入上述三个环节。

## 维护约定

- 新增数据源时，必须同时更新：提供者实现、`INITIAL_CAPABILITIES`、执行器注册表、数据库配置迁移和本文档。
- 免密来源需要明确限定运行环境，并在请求规划中显式声明可无 API Key。
- 修改日线字段、复权方式或来源优先级时，应同步评估 `historicalCoverage` 和缓存导入导出的兼容性。
